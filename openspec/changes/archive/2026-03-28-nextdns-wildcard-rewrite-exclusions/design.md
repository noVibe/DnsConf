## Контекст

Текущий NextDNS-пайплайн читает `BLOCK` и `REDIRECT` из переменных окружения, парсит их как список URL через запятую и затем передаёт обработку redirect-правил через `NextDnsTaskRunner`, `HostsOverrideListsLoader` и `NextDnsRewriteService`. Метод `NextDnsRewriteService.buildNewRewrites()` формирует rewrite-запросы только для точных доменов, а `cleanupOutdated()` удаляет существующие rewrites только в случае, когда тот же домен уже есть, но IP изменился. Сейчас в системе нет отдельного канала конфигурации для wildcard-исключений, нет шага согласования, который удалял бы уже существующие в NextDNS rewrites, попавшие под исключение, и нет Docker-based сценария проверки — в репозитории вообще отсутствуют тесты и контейнерный validation flow.

На дизайн здесь влияют два уже существующих ограничения. Во-первых, проект использует небольшой набор зависимостей и уже содержит Gson, поэтому парсинг новой конфигурации лучше строить на текущем стеке, а не тянуть отдельную библиотеку. Во-вторых, для NextDNS обновление rewrite по сути делается через delete + create, а логика ожидания и повторной попытки при `429` и `524` уже централизована в `NextDnsRateLimitedApiProcessor`. Значит, очистка excluded rewrites должна опираться именно на этот существующий механизм, а не перекладывать расчёт безопасного лимита на пользователя.

## Цели / Не-цели

**Цели:**
- Добавить опциональную GitHub Environment variable для wildcard-исключений NextDNS rewrites в формате JSON, чтобы конфигурировать поведение без изменения кода.
- Исключать совпавшие redirect-домены из создания новых rewrites, не ломая текущее поведение для остальных доменов.
- Сверять уже существующие NextDNS rewrites с exclusion-набором и безопасно удалять совпавшие записи через уже существующий rate-limited API path.
- Добавить Docker-based validation path и placeholder-конфиги, чтобы change можно было проверять и запускать без установки Java на хост-машину.
- Задокументировать, какие значения относятся к GitHub variables, какие — к secrets, и как задаётся новый exclusion JSON.

**Не-цели:**
- Менять поведение Cloudflare или вводить общий wildcard-механизм сразу для обоих DNS-провайдеров.
- Заменять текущую модель `BLOCK` / `REDIRECT` с URL-источниками на полноценную файловую конфигурацию.
- Делать новую отдельную систему rate limiting поверх уже существующего `NextDnsRateLimitedApiProcessor`.
- Требовать реальные NextDNS credentials для обычной unit-test проверки.

## Решения

### 1. Использовать отдельную JSON-переменную окружения для исключений NextDNS rewrites

Change вводит опциональную provider-specific переменную `NEXTDNS_REWRITE_EXCLUSIONS`, значение которой хранится в JSON. Предпочтительный формат:

```json
{
  "patterns": ["*.example.com", "ads.example.org"],
  "cleanupExisting": true
}
```

Поле `patterns` обязательно, если переменная задана. `cleanupExisting` по умолчанию равно `true`. Если переменная отсутствует или пуста, фича считается выключенной и текущее поведение сохраняется. Если же `patterns` заданы, а `cleanupExisting=false`, это отключает только удаление уже существующих excluded rewrites, но не отключает фильтрацию новых rewrite-кандидатов: новые rewrites по-прежнему должны создаваться только для доменов, не совпавших с exclusion patterns. Если JSON некорректный, приложение должно завершаться с понятной ошибкой конфигурации, а не молча игнорировать значение.

**Почему JSON, а не строка через запятую?** JSON позволяет явно описать структуру конфигурации и оставить пространство для будущего расширения без хрупких правил парсинга.

**Альтернатива:** переиспользовать `EnvParser` и для CSV, и для JSON. Отклонено, потому что `EnvParser` сейчас — это минимальный helper для `BLOCK` и `REDIRECT`, и смешивание форматов размоет ответственность и усложнит валидацию.

### 2. Держать parsing и matching исключений в NextDNS-специфичном слое

Wildcard-исключения относятся именно к управлению NextDNS rewrites, поэтому `ListLoader` и `HostsOverrideListsLoader` должны остаться общими и не знать о provider-specific политике. Загрузка redirect-источников остаётся такой, как сейчас: получить данные, нормализовать строки, отфильтровать block-like записи и собрать `BypassRoute`. Применение исключений происходит уже после загрузки источников и до формирования rewrite-запросов — внутри NextDNS rewrite orchestration.

Такой подход не протаскивает NextDNS-логику в общий data-loading слой и не задевает Cloudflare. Поведение остаётся понятным: сначала мы загружаем redirect-кандидаты, затем применяем политику исключений NextDNS, затем сверяемся с фактическим состоянием API.

**Альтернатива:** фильтровать совпавшие домены прямо внутри `ListLoader`. Отклонено, потому что это привязывает общий loader к runtime-конфигурации конкретного провайдера.

### 3. Реализовать wildcard matching небольшим внутренним matcher’ом без новой зависимости

Для этой задачи достаточно поддержки доменных шаблонов с `*`, например `*.example.com` или `ads.*`. Небольшой internal matcher будет приводить домены к lowercase и применять ту же нормализацию ведущего `www.`, которая уже используется в redirect pipeline, а затем сравнивать полную строку. Паттерны будут конвертироваться в anchored regular expression после экранирования литеральных символов, где `*` расширяется в `.*`.

Это оставляет реализацию dependency-free и соответствует текущему стилю проекта.

**Альтернатива:** добавить внешнюю glob/wildcard библиотеку. Отклонено, потому что нужная семантика очень узкая и не оправдывает новую зависимость.

### 4. Явно сверять excluded rewrites и использовать уже существующий rate-limit processor

Во время reconciliation существующих rewrites нужно различать три случая:

1. Существующий rewrite совпадает с exclusion pattern → он попадает в очистку excluded rewrites.
2. Существующий rewrite не совпадает с exclusion pattern, но для того же домена пришёл новый IP → запись удаляется и создаётся заново через текущий outdated-rewrite path.
3. Существующий rewrite уже совпадает с желаемым состоянием → домен убирается из pending create set.

Удаление excluded rewrites должно переиспользовать `NextDnsRateLimitedApiProcessor.callApi(...)`. Сейчас этот processor уже обрабатывает `429` и `524`: возвращает запрос в очередь, ждёт reset interval и продолжает выполнение. Для этой change не нужно заставлять пользователя задавать `maxDeletesPerRun` или другой ручной лимит. Более правильная модель — пытаться удалить все подходящие rewrites через существующий processor и, если NextDNS начнёт throttling, положиться на уже централизованную retry/backoff-логику. Если во время реализации выяснится, что NextDNS использует и другие коды/сигналы для rate limit, расширять нужно именно processor, а не пользовательскую конфигурацию.

**Альтернатива:** вводить configurable per-run deletion cap. Отклонено, потому что пользователь действительно не должен угадывать лимиты провайдера, а проект уже содержит централизованную retry-механику для таких ситуаций.

### 5. Добавить Docker-based validation через Compose и placeholder-файлы конфигурации

Сейчас в репозитории нет ни `src/test`, ни Docker-файлов, ни локального шаблона env-конфига. Change добавляет контейнерный validation entrypoint на базе Docker Compose и официального Java/Maven image, чтобы пользователь мог прогонять вообще все нужные проверки только внутри Docker, без локальной установки Java, Maven и любого другого Java-tooling на хост-машину. При этом сам validation должен оставаться минимальным: не «строить супер-тестовый контур», а проверить основной сценарий фильтрации на реальных hosts-данных из `https://raw.githubusercontent.com/Internet-Helper/GeoHideDNS/refs/heads/main/hosts/hosts` с representative exclusion patterns вроде `*.instagram.com` и `*.facebook.com`, чтобы убедиться, что одни записи отфильтровываются, а другие проходят дальше.

Также change добавляет placeholder-файл вроде `.env.example`, где будут перечислены все входные значения, включая новый `NEXTDNS_REWRITE_EXCLUSIONS`, с явным разделением на GitHub Environment variables и secrets. Для локального запуска пользователь сможет скопировать шаблон, подставить свои данные и не коммитить их в репозиторий.

**Альтернатива:** ограничиться README-командой без Docker-файлов в репозитории. Отклонено, потому что пользователь явно попросил тестирование через Docker, и это должно быть оформлено как first-class workflow.

## Риски / Компромиссы

- **[Некорректный JSON остановит job]** → Это осознанный fail-fast, чтобы не управлять rewrite-состоянием по неверной конфигурации. Риск смягчается документацией и примером конфига.
- **[Семантика wildcard matching может оказаться неочевидной]** → Поддержка остаётся намеренно узкой (`*` only), правила нормализации документируются, а проверка ограничивается минимальным набором сценариев на representative данных.
- **[Большой объём rewrites может заметно увеличить длительность одного запуска]** → Это приемлемо, потому что текущий processor уже умеет ждать reset interval и продолжать выполнение. Нужны хорошие логи прогресса и идемпотентность операций.
- **[Preview-feature настройки Java могут ломать тесты]** → Нужно настроить Maven test execution с `--enable-preview` помимо compile-time флагов, но все такие команды должны выполняться только внутри Docker-контейнера, а не на локальной машине.
- **[Live API tests потребовали бы реальные secrets]** → Базовая verification loop должна оставаться минимальной и опираться на Docker-only checks для фильтрации и сборки, а не на реальный NextDNS аккаунт.

## План миграции

1. Добавить новый config parser, matcher и rewrite reconciliation для опциональной переменной `NEXTDNS_REWRITE_EXCLUSIONS`.
2. Добавить минимальные Docker-runnable тесты/проверки для parsing, matching, exclusion filtering и semantics `cleanupExisting=false`.
3. Добавить Docker Compose validation и placeholder-документацию по переменным окружения так, чтобы ни Java, ни Maven не требовались на хосте.
4. Обновить GitHub Actions / README-документацию, чтобы пользователь мог безопасно задать новый JSON-конфиг.
5. Для rollout достаточно добавить `NEXTDNS_REWRITE_EXCLUSIONS` в GitHub environment `DNS`. Для rollback достаточно убрать эту переменную или временно выставить `cleanupExisting=false`.

## Открытые вопросы

- Нужно ли во время реализации расширить `NextDnsRateLimitedApiProcessor` помимо `429` и `524`, если NextDNS возвращает другие признаки throttling?
- Нужно ли логировать только агрегированные числа очистки, или ещё и печатать конкретные домены перед удалением для более удобной диагностики?
